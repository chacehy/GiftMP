# Etsy Clone — Implementation Plan & Tasks
> Last revised: 2026-05-05

---

## 1. Project Overview

A multi-vendor handmade/vintage marketplace (Etsy-style) where **sellers** create shops, list products, and manage orders, while **buyers** discover, purchase, and review items. The platform earns revenue through a commission on each sale.

### 1.1 Core Value Proposition
- Algerian-focused marketplace with **Chargily** as the local payment gateway.
- Seller storefronts with branding and product catalogs.
- Buyer discovery via search, categories, and curated collections.
- Review & rating system for trust and social proof.

---

## 2. Resolved Decisions

| Decision | Resolution | Rationale |
|---|---|---|
| **Framework** | Next.js 14+ (App Router) | Server Components, Server Actions, streaming, SEO-friendly SSR |
| **Styling** | Tailwind CSS **v3** | Max compatibility with Shadcn UI and BetterAuth UI components |
| **Auth** | BetterAuth | Open-source, DB-agnostic, built-in UI, no vendor lock-in |
| **ORM** | Prisma | Beginner-friendly, readable schema, first-class BetterAuth adapter |
| **Database** | Supabase PostgreSQL (as hosted DB only) | Free tier to start; ORM abstraction enables future migration |
| **Storage** | Supabase Storage (initial) | Simple bucket API; wrapped behind a generic interface for portability |
| **Payments** | Chargily Pay V2 | Algerian market support; REST API + webhook-based confirmation |

> [!IMPORTANT]
> We bypass Supabase Auth and RLS entirely. All access control lives in the Next.js application layer (Server Actions, API Routes, middleware). Supabase is used **only** as a hosted PostgreSQL database and file storage bucket.

## 3. Architecture

```
┌──────────────────────────────────────────────────┐
│                    Client (Browser)               │
│  Next.js App Router — React Server Components     │
│  Tailwind CSS v3 + Shadcn UI                      │
└──────────────┬───────────────────┬────────────────┘
               │                   │
     Server Actions          API Routes
     (mutations)          (/api/auth, /api/webhooks)
               │                   │
       ┌───────┴───────────────────┴───────┐
       │       Next.js Server Layer         │
       │  • BetterAuth (session, RBAC)      │
       │  • Prisma ORM (data access)        │
       │  • Storage wrapper (file uploads)  │
       │  • Chargily SDK (payment init)     │
       └───┬──────────┬──────────┬──────────┘
           │          │          │
   Supabase DB   Supabase    Chargily
   (PostgreSQL)  Storage     Pay API
```

### 3.1 Key Architectural Principles
1. **Server-first**: Mutations via Server Actions; sensitive logic never reaches the client.
2. **ORM abstraction**: Prisma decouples us from Supabase — migration = change one env var.
3. **Generic storage interface**: A `StorageService` wrapper so we can swap Supabase Storage → S3/R2 later.
4. **Webhook-verified payments**: Never trust client-side redirects; confirm via Chargily webhooks.

## 5. Database Schema (Prisma)

### 5.1 Auth Tables (auto-generated by BetterAuth CLI)
BetterAuth manages: `User`, `Session`, `Account`, `Verification`.

### 5.2 Marketplace Tables

```prisma
// ─── Shop ───
model Shop {
  id          String   @id @default(cuid())
  name        String
  slug        String   @unique
  description String?
  bannerUrl   String?
  logoUrl     String?
  userId      String   @unique          // One shop per seller
  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  products    Product[]
  orders      Order[]
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

// ─── Category ───
model Category {
  id        String     @id @default(cuid())
  name      String     @unique
  slug      String     @unique
  imageUrl  String?
  products  Product[]
}

// ─── Product ───
model Product {
  id          String        @id @default(cuid())
  title       String
  description String
  price       Decimal       @db.Decimal(10,2)
  stock       Int           @default(0)
  images      ProductImage[]
  categoryId  String
  category    Category      @relation(fields: [categoryId], references: [id])
  shopId      String
  shop        Shop          @relation(fields: [shopId], references: [id], onDelete: Cascade)
  reviews     Review[]
  cartItems   CartItem[]
  orderItems  OrderItem[]
  isPublished Boolean       @default(false)
  createdAt   DateTime      @default(now())
  updatedAt   DateTime      @updatedAt
}

model ProductImage {
  id        String  @id @default(cuid())
  url       String
  altText   String?
  position  Int     @default(0)
  productId String
  product   Product @relation(fields: [productId], references: [id], onDelete: Cascade)
}

// ─── Cart ───
model Cart {
  id        String     @id @default(cuid())
  userId    String     @unique
  user      User       @relation(fields: [userId], references: [id], onDelete: Cascade)
  items     CartItem[]
  updatedAt DateTime   @updatedAt
}

model CartItem {
  id        String  @id @default(cuid())
  quantity  Int     @default(1)
  cartId    String
  cart      Cart    @relation(fields: [cartId], references: [id], onDelete: Cascade)
  productId String
  product   Product @relation(fields: [productId], references: [id])
  @@unique([cartId, productId])
}

// ─── Order ───
model Order {
  id              String      @id @default(cuid())
  status          OrderStatus @default(PENDING)
  totalAmount     Decimal     @db.Decimal(10,2)
  shippingAddress String
  buyerId         String
  buyer           User        @relation("BuyerOrders", fields: [buyerId], references: [id])
  shopId          String
  shop            Shop        @relation(fields: [shopId], references: [id])
  items           OrderItem[]
  chargilyCheckoutId String?  @unique
  createdAt       DateTime    @default(now())
  updatedAt       DateTime    @updatedAt
}

enum OrderStatus {
  PENDING
  PAID
  PROCESSING
  SHIPPED
  DELIVERED
  CANCELLED
  REFUNDED
}

model OrderItem {
  id        String  @id @default(cuid())
  quantity  Int
  price     Decimal @db.Decimal(10,2)  // Snapshot at time of purchase
  orderId   String
  order     Order   @relation(fields: [orderId], references: [id], onDelete: Cascade)
  productId String
  product   Product @relation(fields: [productId], references: [id])
}

// ─── Review ───
model Review {
  id        String   @id @default(cuid())
  rating    Int                           // 1–5
  comment   String?
  userId    String
  user      User     @relation(fields: [userId], references: [id])
  productId String
  product   Product  @relation(fields: [productId], references: [id], onDelete: Cascade)
  createdAt DateTime @default(now())
  @@unique([userId, productId])           // One review per user per product
}
```

### 5.3 User Extension (added to BetterAuth's User model)
```prisma
// Fields added to the existing User model generated by BetterAuth:
  role    UserRole @default(BUYER)
  shop    Shop?
  cart    Cart?
  reviews Review[]
  buyerOrders Order[] @relation("BuyerOrders")

enum UserRole {
  BUYER
  SELLER
  ADMIN
}
```

## 7. Phased Implementation Plan

### Phase 1 — Foundation & Setup
1. Initialize Next.js project with TypeScript and Tailwind CSS v3.
2. Install and configure Shadcn UI (Button, Card, Input, Dialog, Sheet, etc.).
3. Set up design system: color palette, typography (Inter/Outfit from Google Fonts), spacing tokens.
4. Initialize Prisma ORM, connect to Supabase PostgreSQL, configure `prisma.ts` singleton.
5. Create the full `schema.prisma` (auth + marketplace tables from Section 5).
6. Run `npx prisma migrate dev` to apply the initial migration.

### Phase 2 — Authentication
1. Install `better-auth` and `@better-auth/prisma-adapter`.
2. Configure `lib/auth.ts` with Prisma adapter, email/password provider, and `UserRole` field.
3. Create `app/api/auth/[...all]/route.ts` catch-all handler.
4. Configure `lib/auth-client.ts` for client-side hooks (`useSession`, `signIn`, `signUp`, `signOut`).
5. Build sign-in and sign-up pages with polished UI (Shadcn forms + custom styling).
6. Add auth middleware to protect `/dashboard/*` and `/checkout/*` routes.
7. Create an `AuthProvider` context wrapper in the root layout.
8. Add role selection during sign-up (Buyer vs Seller) or a "Become a Seller" upgrade flow.

### Phase 3 — Layout & Navigation Shell
1. Build the global **Navbar**: logo, search bar, category dropdown, cart icon (with badge), user menu.
2. Build the global **Footer**: links, social icons, copyright.
3. Build the **Seller Dashboard Sidebar**: navigation for Shop, Products, Orders, Settings.
4. Implement responsive behavior (mobile hamburger menu, collapsible sidebar).

### Phase 4 — Seller Experience
1. **Shop Setup**: Form to create/edit shop (name, slug, description, logo, banner image upload).
2. **Storage wrapper** (`lib/storage.ts`): generic `uploadFile` / `deleteFile` / `getPublicUrl` functions backed by Supabase Storage.
3. **Product CRUD**:
   - Product creation form: title, description, price, stock, category select, multi-image upload, publish toggle.
   - Product listing table in dashboard with edit/delete actions.
   - Server Actions in `actions/product.actions.ts` with Zod validation.
4. **Order Management**: Table of incoming orders with status badges; action to update status (Processing → Shipped → Delivered).
5. **Seller Dashboard Home**: Stats cards (total revenue, total orders, total products, average rating).

### Phase 5 — Buyer Experience
1. **Home Page**:
   - Hero section with dynamic headline, CTA, and background imagery.
   - Featured categories carousel/grid.
   - Trending products section.
   - How-it-works section.
   - Micro-animations (fade-in on scroll, hover scales on cards).
2. **Product Discovery**:
   - `/products` page with server-side search + filtering (category, price range, sort by).
   - Responsive product grid using `ProductCard` component.
   - Pagination or infinite scroll.
3. **Product Detail** (`/product/[id]`):
   - Image gallery with thumbnail navigation.
   - Product info (title, price, description, stock indicator).
   - "Add to Cart" button with quantity selector.
   - Shop info card (link to seller's storefront).
   - Reviews section with average rating display and review list.
   - Review submission form (authenticated buyers only, post-purchase).
4. **Seller Storefront** (`/shop/[slug]`):
   - Shop banner, logo, description.
   - Grid of the seller's products.
5. **Shopping Cart** (`/cart`):
   - Cart items list with quantity adjustment and remove button.
   - Order summary sidebar (subtotal, estimated total).
   - "Proceed to Checkout" CTA.
6. **Checkout** (`/checkout`):
   - Shipping address form.
   - Order review.
   - "Pay with Chargily" button → calls Server Action → redirects to Chargily checkout URL.
7. **Buyer Order History** (`/orders`):
   - List of past orders with status, date, and total.
   - Order detail view with item breakdown.

### Phase 6 — Payments (Chargily Integration)
1. Install `@chargily/chargily-pay` SDK.
2. Create `lib/chargily.ts` helper for initializing checkouts.
3. Create `/api/webhooks/chargily/route.ts`:
   - Verify webhook signature.
   - On `checkout.paid`: update Order status to `PAID`, decrement product stock.
   - On `checkout.failed`: update Order status to `CANCELLED`.
4. Configure success/failure redirect URLs.
5. Test full payment cycle in Chargily test mode.

### Phase 7 — Messaging System (MVP)
1. Create `Message` and `Conversation` models in Prisma.
2. Inbox UI for buyers and sellers.
3. Conversation thread view.
4. Server Actions for sending/receiving messages.

### Phase 8 — Admin Panel (Post-MVP)
1. Admin dashboard with platform-wide stats.
2. User management (view, suspend, delete).
3. Product moderation (flag, review, remove).
4. Commission tracking and reporting.

### Phase 9 — Polish, SEO & Performance
1. Add `<title>`, `<meta description>`, Open Graph tags to every page using Next.js `metadata` API.
2. Use semantic HTML (`<main>`, `<article>`, `<section>`, `<nav>`, `<footer>`).
3. Optimize images with `next/image` and proper `alt` text.
4. Implement loading skeletons and `Suspense` boundaries.
5. Lighthouse audit targeting 90+ on Performance, Accessibility, SEO, Best Practices.
6. Full responsive QA on mobile, tablet, and desktop viewports.

### Phase 10 — Migration Strategy (Scaling)
1. **Database**: Change `DATABASE_URL` to new PostgreSQL host (Neon, AWS RDS, etc.). Run `prisma migrate deploy`. Use `pg_dump`/`pg_restore` for data.
2. **Storage**: Bulk-transfer Supabase bucket → S3/R2/Spaces. Update `lib/storage.ts` to use new provider. Update stored URLs in DB via migration script.
3. **No auth changes needed** — BetterAuth is DB-agnostic via Prisma adapter.
